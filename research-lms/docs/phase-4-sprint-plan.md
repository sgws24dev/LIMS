## Architecture Decisions

### Service Boundary
Phase 4 introduces **three new microservices**, each following Clean Architecture (Domain, Application, Infrastructure, Api):

| Microservice | Modules | Rationale |
|-------------|---------|-----------|
| `Services/ServiceWorkflow/` | Form Builder + Service Requests + Workflow Engine | Tightly coupled: requests use forms, forms drive workflows |
| `Services/Projects/` | Projects + Work Orders + Cost Centers + Issues | Project lifecycle naturally includes issues and work orders |
| `Services/Inventory/` | Inventory + Vendors + Purchase Orders | Supply chain lifecycle with independent scaling needs |

### Shared Kernel Dependencies
Cross-cutting types live in `ResearchLms.Shared`:
- `BaseEntity`, `IEvent`, `ValueObject` (already exist)
- `ResourceType` enum (shared with Scheduling/Facility)
- `ITenantContext` for multi-tenancy
- `IDateTimeProvider` for testable time
- `INotificationService` for email/SMS (interface only; implementation in Phase 7)

### Form Definition Strategy
Forms use a **JSON Schema** approach:
- `FormDefinitions` stores the schema as a JSON blob (compliant with JSON Schema Draft 2020-12)
- Dynamic form rendering on frontend reads the schema and generates fields
- No code generation вҖ” forms are fully data-driven
- Supports: text, number, date, select, multi-select, file upload, signature, repeatable sections

### Workflow State Machine
Workflows use a **simple state machine** pattern (no external library):
```
Draft вҶ’ Submitted вҶ’ UnderReview вҶ’ Approved вҶ’ InProgress вҶ’ Completed
                   вҶ’ RevisionRequested вҶ’ Draft (loop)
                                          вҶҳ Rejected
```
- `WorkflowDefinitions` stores states, transitions, guards, and actions
- `WorkflowInstances` tracks current state + history
- Transitions can trigger MassTransit events for side effects (notifications, SLA timers)

### Project вҶ’ Work Order вҶ’ Issue Relationship
```
Project
  в”ңв”Җв”Җ WorkOrders (scheduled maintenance, tasks)
  в”Ӯ     в””в”Җв”Җ Issues (problems discovered during work)
  в””в”Җв”Җ CostCenters (budget tracking)
```
- A Project can have multiple Work Orders
- A Work Order can reference multiple Issues
- An Issue can optionally be linked to a Work Order or stand alone
- Cost Centers aggregate spend across Projects and Work Orders

### Inventory Strategy
- `InventoryItems` tracks stock levels with min/max thresholds
- Reorder alerts are generated by a Hangfire recurring job (hourly check)
- `PurchaseOrders` tracks the procurement lifecycle: Draft вҶ’ Submitted вҶ’ Approved вҶ’ Ordered вҶ’ Received
- Barcode scanning via `IBarcodeScanService` abstraction (concrete implementation uses ZXing.Net)
- Vendor management with pricing agreements and lead times

### Data Isolation
Each microservice gets its own DbContext with its own connection string:
- `ServiceWorkflowDbContext` вҶ’ `ServiceWorkflowDb`
- `ProjectsDbContext` вҶ’ `ProjectsDb`
- `InventoryDbContext` вҶ’ `InventoryDb`

Multi-tenancy is enforced via the same `ITenantContext` + global query filter pattern.

### Frontend Routing
New frontend modules:
- `modules/requests/` вҶ’ `/requests/*`
- `modules/workflow/` вҶ’ `/workflow/*`
- `modules/projects/` вҶ’ `/projects/*`
- `modules/issues/` вҶ’ `/issues/*`
- `modules/inventory/` вҶ’ `/inventory/*`

| `Approvals` | TenantId, ServiceRequestId, ApproverId, Status, Comment, AssignedAt, DecidedAt | Linked to service request |

| `WorkflowInstances` | TenantId, WorkflowDefinitionId, EntityType, EntityId, CurrentState, StateHistory (nvarchar(max) вҖ” JSON), ContextData (nvarchar(max) вҖ” JSON), Status | EntityType+EntityId for polymorphic reference |

## Dependencies & Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Workflow execution engine with dynamic guard evaluation is complex | High | Start with simple hardcoded guards (role check, field value comparison). Add expression-based evaluation in a later iteration. The state machine itself is straightforward. |
| ServiceNow/Jira API may not be available for testing | Medium | Build `IIssueSyncService` with a configurable HTTP endpoint. Provide a mock implementation for dev/test. Real connector can be added per-customer. |
| Form Builder drag-and-drop designer is frontend-heavy | Medium | Use react-beautiful-dnd for field ordering. For initial release, skip the visual designer and provide a JSON editor + preview. Designer can be Sprint 2 if time runs short. |
| Inventory barcode scanning needs camera/webcam access | Low | Support file upload as fallback. ZXing.Net handles decoding from uploaded images without camera access. |
| Three new microservices increase DevOps complexity | Medium | All three share the same Docker Compose setup. Add them incrementally (ServiceWorkflow in Sprint 1, Projects in Sprint 3, Inventory in Sprint 4). Each follows identical patterns from existing Scheduling service. |
| Dynamic form JSON schema validation may have edge cases | Medium | Use a well-tested library (JsonSchema.NET). Add comprehensive unit tests for schema validation. Start with basic field types and add complex ones (repeatable sections, conditional visibility) in later sprints. |
| No notification templates exist yet (Phase 7) | Low | For Phase 4, embed notification text directly in the workflow/approval configuration. A Phase 7 migration can refactor to use the template system. |
